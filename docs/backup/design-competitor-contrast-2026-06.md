# Phase 1 — 경쟁 제품 UI/UX 대조 (2026-06)

> 목적: 웹소설 작가용 집필 워드프로세서들의 UI/UX·실사용 후기를 조사하고
> Luie 현재 상태(특히 `MainLayout`의 sidebar/toolbar/binder/footer 시각 분열)와 대조한다.
> Luie 이념: **"집필을 방해하지 않는다"** — 이 한 줄을 기준으로 모든 제품을 줄세운다.

## 2. 후기에서 반복되는 신호 (작가들이 실제로 말하는 것)

1. **몰입(distraction-free)이 거의 모든 호평의 중심축.**
   스크리브너조차 가장 많이 칭찬받는 단일 기능이 "모든 UI를 가리는 Composition Mode". 율리시스·iA Writer는 그걸 *기본값*으로 만들어 사랑받는다.

2. **"통합감"이 차별화 포인트.**
   펜시브의 차별점은 기능이 아니라 **"여러 도구의 기능을 깔끔하게 하나로 통합한 느낌"**. 작가는 도구가 여러 앱을 짜깁기한 것처럼 보이는 걸 싫어한다.

3. **복잡함·학습곡선은 가장 흔한 이탈 사유.**
   스크리브너의 변치 않는 단점 = "기능은 좋은데 복잡하고 배우는 데 오래 걸린다". 시각적 복잡함도 같은 범주의 비용으로 읽힌다.

4. **색감 = 피로의 직접 원인.**
   다크모드라도 *너무 어두운 배경 + 밝은 글씨*는 "눈뽕"·피로 호소. 글자색을 형형색색으로 쓰는 것도 피로. → **저대비·차분·일관된 톤**이 장시간 집필에 유리.

5. **안정감(자동저장·조용한 피드백).**
   뮤블의 "블록 저장 실패" 빨강 경고가 공포로 회자됨. 작가는 시끄러운/불안한 신호를 싫어한다.

## 3. Luie 현재 상태와의 대조 (핵심)

스크린샷 + 기존 진단(`design-audit-2026-06.md`) 기준, `MainLayout` 한 화면 안에서:

| 영역 | 현재 시각 처리 | 문제 |
|---|---|---|
| 좌측 **sideBar** (binder) | `bg-sidebar`, 다크, 행마다 아이콘+들여쓰기 | 그 자체는 OK |
| 상단 **ToolBar** (`EditorToolbar.tsx`) | 별도 채도/테두리/컨트롤 곡률 | 사이드바·본문과 다른 "스킨" |
| 본문 에디터 | `bg-app` | 조용함(좋음) |
| **binderSideBar** | bg가 `bg-sidebar`/`bg-panel`/`bg-surface`로 혼재 | sidebar와 외관 구분 안 됨 |
| 하단 **footer** (저장됨/글자수/내보내기) | 또 다른 톤 | 4번째 "다른 앱" |

→ **결론: Luie는 지금 "스크리브너의 단점(복잡·분열)"을 닮아 있고, "율리시스/펜시브의 강점(통합·미니멀·조용함)"과 정반대다.** 이념("방해하지 않는다")과 실제 화면이 직접 충돌한다.

### 불쾌한 골짜기의 정체
하나의 화면에 **4개 영역이 각자 다른 채도/명도/테두리/곡률 언어**를 써서, 사용자 눈에는 "한 제품의 4영역"이 아니라 "붙여놓은 4개 패널"로 읽힌다. 이건 색 *값*의 문제 이전에 **영역 간 시각 대비가 통제되지 않은** 문제다.

## 4. Phase 1이 도출한 방향 (다음 Phase 입력값)

경쟁 제품이 입증한 원칙을 Luie 이념에 맞춰 정렬하면:

1. **영역 간 대비 최소화 (통합감).** sidebar·toolbar·binder·footer가 *같은 배경 패밀리* 안에서 미세한 단계차(elevation)로만 갈리도록. 테두리/그림자로 영역을 "자르지" 말 것. → 율리시스/펜시브 통합감.
2. **toolbar·footer는 "조용한 크롬"으로.** 본문이 주인공. 주변 크롬은 저대비 회색 톤, hover 시에만 또렷해지도록. → distraction-free.
3. **저대비 차분한 다크 톤.** 순수 검정+고대비 흰글씨 회피. 영역 구분은 명도 1~2단계 차이로만.
4. **한 가지 시각 언어(곡률/구분선/간격) 공유.** 이미 확정된 역할→토큰 컨벤션(rail/binder=`bg-sidebar`, editor=`bg-app`, inspector=`bg-panel`, modal=`bg-surface`)을 toolbar/footer까지 확장.

> 다음 Phase: 위 4원칙을 `MainLayout` + `EditorToolbar` + footer(StatusFooter) + binder에 적용하는 구체 토큰 설계.
