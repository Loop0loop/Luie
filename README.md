## 2. Project Overview

Luie는 장편 집필 과정에서 반복되는 "원고, 설정, 백업, 출력물이 서로 흩어지는 문제"를 줄이기 위해 만든 Electron 데스크톱 앱이다.
웹소설 작업은 단순히 글을 쓰는 것이 아니라, 챕터, 캐릭터, 용어, 사건, 관계, 스냅샷, 내보내기를 함께 관리해야 한다.
이 프로젝트는 그 흐름을 프로젝트 단위로 묶고, 로컬 저장과 복구를 중심에 두는 방식으로 해결한다.
즉, 문서를 옮겨 다니는 도구가 아니라 창작 흐름 자체를 하나로 정리하는 작업 공간을 목표로 한다.

## 3. Install / Run

### Requirements

- Bun 1.3.10
- Node.js 환경에서 Electron 관련 네이티브 빌드가 가능한 상태
- Git

### Install

```bash
git clone <repository-url>
cd Luie
bun install
```

첫 설치 시에는 `postinstall` 단계에서 Prisma 생성, Electron 재빌드, 네이티브 haptics 빌드가 함께 수행된다.
그래서 첫 실행은 일반적인 웹 프로젝트보다 조금 더 오래 걸릴 수 있다.

### Run

```bash
bun dev
```

### Build

```bash
bun run build
```

### Validation

```bash
bun run typecheck
bun run qa:core
```

## 5. Features & Examples

### 프로젝트 시작 비용을 줄인다

- 템플릿 기반 프로젝트 생성으로 초반 세팅 부담을 줄인다.
- 예시:
  - `Blank`: 바로 자유 서술을 시작한다.
  - `Novel`: 웹소설 문법에 맞는 기본 구조로 시작한다.
  - `Script`: 대본 형식에 맞춰 시작한다.
  - `Essay`: 일반 문서형 작업을 시작한다.

### 원고와 세계관을 분리하지 않는다

- 원고 작성과 설정 관리가 같은 프로젝트 안에서 이어진다.
- 예시:
  - 챕터를 쓰다가 캐릭터를 추가한다.
  - 용어를 정리하면서 사건과 파벌을 연결한다.
  - 그래프 뷰에서 관계를 확인하고, 다시 에디터로 돌아온다.

### 저장 실패를 작업 손실로 바꾸지 않는다

- 자동 저장, 스냅샷, 복구 후보, 비상 저장을 함께 둬서 실패 지점을 줄인다.
- 예시:
  - 작업 중 렌더러가 불안정해져도 메인 프로세스가 복구를 시도한다.
  - 복원 가능한 `.luie` 파일이 있으면 다시 열어 이어서 작업한다.

### 출력 전에 품질을 확인한다

- 내보내기 전에 미리보기를 통해 레이아웃을 먼저 확인한다.
- 예시:
  - 페이지 번호가 맞는지 확인한다.
  - 여백과 줄간격이 제출 기준에 맞는지 확인한다.
  - DOCX와 HWPX를 상황에 맞게 선택한다.

### AI 분석과 로컬 폴백을 함께 둔다

- 원고 분석은 AI 스트리밍을 사용하지만, 실패 시 로컬 규칙 기반 분석으로 이어진다.
- 예시:
  - 외부 API가 불안정하면 로컬 분석으로 핵심 흐름을 먼저 보여준다.
  - 분석 결과는 인용문과 함께 보여줘서 원고의 어느 부분에서 나온 판단인지 확인할 수 있다.

## 6. Problem Solving

### 문제 1. 작업 정보가 여러 파일과 화면으로 흩어진다

- 문제
  - 원고, 세계관, 최근 프로젝트, 첨부 경로가 분리되면 작업 맥락이 자주 끊긴다.
- 왜 생겼는지
  - 장편 집필은 단일 문서가 아니라 프로젝트 단위 상태 관리가 필요하다.
  - 웹앱 방식만으로는 로컬 파일, 네이티브 대화상자, 패키지 복구를 일관되게 다루기 어렵다.
- 어떻게 해결했는지
  - Electron 구조에서 메인 프로세스가 파일, DB, 패키지, 복구를 책임지도록 분리했다.
  - `ProjectService`와 SQLite + Prisma를 중심 저장소로 두고, `.luie` 패키지와 첨부 경로를 함께 정리했다.
  - Preload를 통해 Renderer가 직접 OS 자원에 접근하지 않도록 경계를 분리했다.
- 결과
  - 프로젝트가 커져도 "어디에 뭐가 있는지"를 기억하는 부담이 줄고, 다시 열었을 때 흐름을 복원하기 쉬워졌다.

### 문제 2. 저장 실패 한 번이 긴 작업을 망칠 수 있다

- 문제
  - 장편 원고는 수정량이 많아서 저장 실패가 곧 데이터 손실로 이어질 수 있다.
- 왜 생겼는지
  - 자동 저장만으로는 렌더러 충돌, 파일 손상, 패키지 복구 실패까지 모두 커버하기 어렵다.
- 어떻게 해결했는지
  - 자동 저장 미러, 스냅샷, 복구 후보, 비상 저장 파일, 시작 시 정리 작업을 분리했다.
  - 렌더러 충돌 시에는 메인 프로세스에서 critical flush를 먼저 수행하도록 했다.
- 결과
  - 단순 저장 기능이 아니라, 실패 이후에도 다시 이어 쓰는 복구 경로를 갖게 됐다.

### 문제 3. 세계관 정보가 원고와 따로 놀면 연속성이 깨진다

- 문제
  - 캐릭터, 용어, 사건, 파벌, 관계가 따로 놀면 설정 충돌을 사람이 전부 기억해야 한다.
- 왜 생겼는지
  - 챕터가 쌓일수록 참조 관계가 급격히 복잡해지고, 설정 오류가 누적된다.
- 어떻게 해결했는지
  - Terms, Synopsis, Mind Map, Drawing, Plot, Graph를 하나의 World 섹션으로 묶었다.
  - World Graph는 엔티티 선택과 워크스페이스 상태를 동기화하고, 고아 관계 정리 서비스와 연결된다.
- 결과
  - 세계관 정보가 단순 메모가 아니라, 원고와 연결된 작업 자산이 되었다.

### 문제 4. 내보내기는 마지막 버튼이 아니라 품질 검수 단계다

- 문제
  - 출력 결과물이 기대한 포맷과 다르면 다시 전체를 손봐야 한다.
- 왜 생겼는지
  - 제출용 문서는 여백, 줄간격, 글꼴, 페이지 번호 같은 요소가 민감하다.
- 어떻게 해결했는지
  - Export Window에서 DOCX/HWPX 미리보기와 설정 패널을 분리했다.
  - HTML 원고를 문단 단위로 정규화하고, 출력 품질을 조정할 수 있게 했다.
- 결과
  - 최종 파일을 다시 만드는 비용을 줄이고, 출력 전 검수 흐름을 넣을 수 있었다.

### 기술적 판단

- **왜 Electron?**
  - 로컬 파일, 네이티브 대화상자, 복구 파일, 멀티 윈도우 흐름을 모두 다뤄야 해서 데스크톱 컨테이너가 맞았다.
- **왜 SQLite + Prisma?**
  - 오프라인 중심 저장, 단일 사용자 프로젝트 관리, 타입 안전한 스키마 접근이 잘 맞았다.
- **왜 Main / Preload / Renderer 분리?**
  - 렌더러는 UI에 집중하고, 지속성 있는 작업은 메인에서 처리해 안정성과 테스트 가능성을 높이기 위해서다.
- **왜 contract 테스트를 둔 이유?**
  - IPC, store, persist 경계를 명시적으로 검사해 변경 충돌을 줄이기 위해서다.

## 7. Performance / Validation

- `bun run qa:core`
  - typecheck, lint, IPC contract, store usage, persist contract, core complexity, build warning 회귀를 함께 점검한다.
- `tests/e2e/smoke.spec.ts`
  - 앱이 템플릿 선택 화면까지 정상 부팅되는지 확인한다.
- `tests/e2e/e2e.spec.ts`
  - 프로젝트 생성, 챕터 생성, 검색, 스냅샷 복원, 자동 저장 흐름을 검증한다.
- `tests/e2e/stress.spec.ts`
  - IPC로 챕터 200개를 연속 생성하는 스트레스 시나리오를 검증한다.
- `tests/main/services/projectService.test.ts`
  - CRUD, `.luie` 복구, 대용량 export 안정성을 확인한다.
- `tests/main/services/snapshotService.test.ts`
  - 스냅샷 생성, 복원, 비상 저장 흐름을 확인한다.
- `tests/main/services/syncService.test.ts`
  - 동기화 인증과 상태 전이를 검증한다.
- `tests/e2e/visual.spec.ts`
  - 템플릿 선택 화면과 에디터의 시각 회귀를 잡는다.

### 성과 메모

- `200챕터` 스트레스 테스트가 실제로 존재한다.
- `5,000,000자` 규모의 world payload에 대한 export 안정성 테스트가 있다.
- `5만자 이상` 원고 처리와 `100챕터+5만자` 스트레스 시나리오가 테스트 문서에 포함돼 있다.
